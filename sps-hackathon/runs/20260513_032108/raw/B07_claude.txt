# Software Specification: User Activity Logging System

## 1. Core Behavior

The system must record a structured log entry for every significant user action performed within the application. Logging must occur automatically, transparently to the user, and without interrupting normal system operation. Log entries must be persisted durably, queryable by authorized personnel, and retained for a defined period.

### 1.1 Logged Event Categories

The following categories of user activity must be captured:

| Category | Examples |
|---|---|
| Authentication | Login, logout, failed login, password reset, MFA challenge |
| Authorization | Permission denied, role change, privilege escalation attempt |
| Data Access | Read, search, export of sensitive or protected records |
| Data Mutation | Create, update, delete of any persistent resource |
| Configuration Changes | System settings, feature flags, user preferences |
| Session Events | Session creation, expiration, forced termination |
| Administrative Actions | User creation, account suspension, audit log access |
| File Operations | Upload, download, delete of files or attachments |
| API Usage | External API calls made on behalf of a user |

### 1.2 Logging Trigger

A log entry is generated:
- Immediately upon completion of a user action (post-execution).
- Immediately upon failure or rejection of a user action (e.g., authorization denial).
- Before execution for destructive or irreversible operations (pre-execution record), with a follow-up entry indicating outcome.

## 2. Inputs

Each log entry is constructed from the following data sources at the time of the event:

### 2.1 Mandatory Fields

| Field | Type | Description |
|---|---|---|
| `event_id` | UUID v4 | Globally unique identifier for this log entry |
| `timestamp` | ISO 8601 UTC | Exact date and time the event occurred |
| `user_id` | String | Unique identifier of the acting user |
| `session_id` | String | Identifier of the active session |
| `action` | Enum | Standardized action code (e.g., `USER_LOGIN`, `RECORD_DELETE`) |
| `resource_type` | String | Type of resource acted upon (e.g., `Invoice`, `UserAccount`) |
| `resource_id` | String | Identifier of the specific resource instance |
| `outcome` | Enum | `SUCCESS`, `FAILURE`, `DENIED`, `ERROR` |
| `ip_address` | String | IP address of the client at time of request |
| `user_agent` | String | HTTP User-Agent string of the client |

### 2.2 Optional / Contextual Fields

| Field | Type | Description |
|---|---|---|
| `previous_value` | JSON | Snapshot of resource state before mutation |
| `new_value` | JSON | Snapshot of resource state after mutation |
| `error_code` | String | Application or HTTP error code on failure |
| `error_message` | String | Human-readable error description |
| `geo_location` | String | Derived country/region from IP (best-effort) |
| `request_id` | String | Correlation ID linking to the originating HTTP request |
| `actor_role` | String | Role(s) held by the user at time of action |
| `delegated_by` | String | User ID of an admin acting on behalf of another user |
| `tags` | Array[String] | Arbitrary labels for filtering (e.g., `["pii", "financial"]`) |

---

## 3. Outputs

### 3.1 Persistent Log Store

- Each log entry is written to a dedicated, append-only log store (separate from the application's primary database).
- Log entries are stored in structured JSON format.
- The store must support indexed queries on: `user_id`, `timestamp`, `action`, `resource_type`, `resource_id`, `outcome`, `ip_address`, and `session_id`.

### 3.2 Query / Export API

An internal API (not exposed publicly) must allow authorized administrators to:
- Query logs by any indexed field or combination thereof.
- Filter by date range (inclusive).
- Paginate results (default page size: 100; maximum: 1000).
- Export results as JSON or CSV.
- Receive results sorted
