Specification for API Endpoint Returning HTTP 404 with JSON Error Body:

Core Behavior:
- The API endpoint must detect when a requested resource does not exist and respond with an HTTP 404 status code.
- The response must include a JSON-formatted error body containing an error code and a human-readable message.

Inputs:
- The API endpoint accepts HTTP requests that specify a resource identifier (e.g., a URL path or query parameter indicating the resource being requested).

Outputs:
- The API endpoint returns an HTTP response with:
  - Status code: 404 Not Found
  - Content-Type: application/json
  - JSON error body containing:
    - "error_code": A string or numeric code uniquely identifying the error type (e.g., "RESOURCE_NOT_FOUND").
    - "message": A human-readable string providing details about the error (e.g., "The requested resource does not exist.").

Assumptions:
- The error code "RESOURCE_NOT_FOUND" will be used as a standard identifier for this specific error condition.
- The human-readable message will be a generic statement indicating the non-existence of the resource.
- The API endpoint is assumed to be part of a RESTful service.
- The resource identifier is assumed to be a valid input format (e.g., a valid UUID or integer ID).

Failure Modes:
- If the resource identifier is malformed or invalid, the system should handle this as a separate error condition, potentially returning a different status code (e.g., 400 Bad Request) with an appropriate error message.
- If the system encounters an internal error while processing the request, it should return a 500 Internal Server Error with a relevant error message, not a 404.

Edge Cases:
- If the resource identifier is syntactically correct but semantically invalid (e.g., an ID that does not correspond to any resource), the system should still return a 404 status with the specified error body.
- If the request is made with an unsupported HTTP method (e.g., POST instead of GET), the system should return a 405 Method Not Allowed, not a 404.

Security Constraints:
- The system must ensure that no sensitive information about the resource or the system is exposed in the error message.
- The system should validate the resource identifier to prevent injection attacks or other security vulnerabilities.

Performance Constraints:
- The system should be able to detect and respond with a 404 error within a reasonable time frame, ideally under 100 milliseconds, to ensure a responsive user experience.

This specification outlines the expected behavior and constraints for the API endpoint when handling requests for non-existent resources.
