# Software Specification: Conversion-Optimized Checkout Flow

## 1. Core Behavior

### 1.1 Checkout Flow Architecture
The checkout flow shall be implemented as a **linear, progressive multi-step process** with a maximum of **4 steps**: Cart Review → Contact & Shipping → Payment → Order Confirmation. Steps shall be presented as a persistent progress indicator visible at all times during checkout.

### 1.2 Guest Checkout
The system shall allow users to complete a purchase without creating an account. Account creation shall be offered **only after** order confirmation, never as a gate before purchase.

### 1.3 Persistent Cart
Cart contents shall persist across sessions for a minimum of **30 days** using a combination of cookies and server-side storage for authenticated users, and cookies/localStorage for guests.

### 1.4 Inline Validation
All form fields shall validate in real time (on blur) and provide immediate, field-level feedback without requiring form submission. Errors shall be resolved inline without clearing other field values.

### 1.5 Auto-Fill and Smart Defaults
The system shall support browser autofill (via correct `autocomplete` attributes), address lookup via postal code, and pre-population of fields for returning users. Country shall default to the user's detected locale.

### 1.6 Order Summary Visibility
A collapsible order summary (expanded by default on desktop, collapsed by default on mobile) shall be visible on every checkout step, showing item thumbnails, quantities, line prices, subtotal, shipping estimate, taxes, and total.

### 1.7 Express Checkout
The system shall surface express checkout options (e.g., Shop Pay, Apple Pay, Google Pay, PayPal) at the top of the checkout flow, before the standard form, on Step 1 (Cart Review) and the Payment step.

## 2. Inputs

### 2.1 User-Provided Data

| Field | Type | Required | Validation Rules |
|---|---|---|---|
| Email address | String | Yes | RFC 5322 format; real-time duplicate account detection with login prompt |
| First name | String | Yes | 1–50 chars, Unicode letters and hyphens |
| Last name | String | Yes | 1–50 chars, Unicode letters and hyphens |
| Phone number | String | Optional (required for SMS updates if opted in) | E.164 format; locale-aware formatting hint |
| Shipping address line 1 | String | Yes | 5–100 chars |
| Shipping address line 2 | String | No | 0–100 chars |
| City | String | Yes | 2–50 chars |
| State/Province | String | Yes | Validated against country-specific subdivision list |
| Postal/ZIP code | String | Yes | Validated against country-specific format regex |
| Country | String | Yes | ISO 3166-1 alpha-2; dropdown |
| Shipping method selection | Enum | Yes | Derived from available carrier options |
| Payment method | Enum | Yes | Credit/debit card, express wallets, buy-now-pay-later |
| Card number | String | Yes (if card) | Luhn algorithm check; PAN never stored on application servers |
| Card expiry | MM/YY | Yes (if card) | Must be a future date |
| CVV | String | Yes (if card) | 3–4 digits; never logged or stored |
| Billing address | String | Conditional | Defaults to "same as shipping"; full address required if different |
| Promo/coupon code | String | No | Applied in real time; clear success/failure feedback |

### 2.2 System-Provided Inputs
- Session/authentication token
- Cart state (items, quantities, prices) from cart service
- Shipping rate quotes from carrier integration
- Tax calculation from tax service (address-triggered)
- Fraud risk score from fraud detection service
